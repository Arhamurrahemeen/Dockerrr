## Voice (when writing any prose — README, comments meant to be read, commit messages that matter)

- First person, no hedging. Confident, direct, occasionally blunt.
- Contractions always.
- Pakistan context (load-shedding, Urdu UI, low-bandwidth, Mirpurkhas/Hyderabad/Karachi) is a normal constraint, never flagged as cultural color.
- Personal anchors over abstract claims — real project names, real people, real numbers when available. Never invent metrics or outcomes — flag placeholders instead of filling them in.
- Avoid: leverage, unlock, 10x, synergy, journey, ecosystem, holistic, hedging phrases, "🚀 Excited to announce" energy.

## How Arham likes to work

- **Phase-by-phase with greenlights.** Build in stages, stop, wait for approval. A one-word "go" means proceed.
- **Fork choices, not open questions.** Real decisions get 2–3 labeled options with a one-line tradeoff each, not an open-ended "what do you think?"
- **Full push over incremental.** Default to the complete, working thing over a rough draft, once the direction is confirmed.
- **Full context expected.** Don't make him re-explain conventions already established in this repo or in past sessions.
- **Push back early** if the framing or approach is wrong — don't execute a bad brief politely.

## Repo structure conventions

- Folders are by *type of thinking*, never topic: `playbooks/`, `philosophies/`, `career-growth/`, `case-studies/`, `tool-stacks/`, `field-notes/`, `builds/`, `assets/`.
- `builds/` — fast, fun, standalone technical projects (Docker experiments, quick builds). Distinct from the reflective/methodology folders. Each build gets its own subfolder, its own README following the lightweight build-README pattern (see `builds/multiplayer-snake-lan/README.md` for the template once it exists).
- Filenames: lowercase-kebab-case. Assets in `assets/`, named `<piece-slug>-<purpose>.svg`.
- Visual system for any generated SVG/diagram: cream `#F4EEE2`, ink `#1C1B19`, teal `#539091`, muted `#8A8275`, white cards with 5px teal left accent bar, Georgia italic for display type, Courier New for labels. Full spec in `CONTEXT.md` §4.
- Method over tools: durable methodology is the spine of any written piece; specific tools go in a dated "current stack" callout, not woven into the method itself.

## What not to do

- Don't scaffold generic tutorial-style projects — every build should have a reason it's *this* build and not a docs example.
- Don't invent data, metrics, or screenshots. Flag what's missing instead.
- Don't treat this repo and the LinkedIn/brand side as separate concerns when writing anything meant to be published — same voice, same spine, different delivery format.
